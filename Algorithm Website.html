my_borrowing_history() & my_returned_books()

Purpose: Display historical borrowing data with custom sorting.

üîÑ Custom Sort Algorithm

  1. Convert return dates to timestamps using strtotime()
  2. Compare return dates:
    • If different: Sort by return_date DESC (most recent first)
    • Uses spaceship operator <=>:
      • Returns -1 if A < B (A before B)
      • Returns 0 if A = B
      • Returns 1 if A > B (A after B)
  3. If return dates are equal:
    • Sort by updated_at DESC (most recently updated first)
  4. Reindex collection with values()
$sortedHistory = $processedHistory->sort(function ($a, $b) { $returnA = strtotime($a['return_date']); $returnB = strtotime($b['return_date']); if ($returnA !== $returnB) { return $returnB <=> $returnA; // DESC by return date } return $b['updated_at'] <=> $a['updated_at']; // DESC by updated_at })->values();

üéØ Sorting Algorithms Summary

Spaceship Operator (<=>)

Used for three-way comparison in custom sorting functions.

  • a <=> b returns:
    • -1 if a < b
    • 0 if a = b
    • 1 if a > b
  • DESC sorting: b <=> a (reverse order)
  • ASC sorting: a <=> b (natural order)

Priority-Based Sorting

Used in loan management to prioritize urgent items.

  1. Primary Key: Most important criteria (e.g., due date)
  2. Secondary Key: Tiebreaker when primary keys are equal
  3. Implementation: Chain orderBy() clauses or use custom comparison functions

üîß Filtering Techniques

LIKE Pattern Matching

Flexible text search across multiple fields.

WHERE field LIKE '%search_term%'
  • % = wildcard (matches any characters)
  • Finds partial matches anywhere in the text
  • Combined with OR for multi-field search

Relationship-Based Filtering

Filter parent models based on related model criteria.

  • whereHas(): Filter by existence of related records matching criteria
  • orWhereHas(): OR condition for relationship filtering
  • with(): Eager load relationships to avoid N+1 queries
  • withCount(): Load count of related records

Subquery Filtering

Calculate availability using raw SQL subqueries.

Pattern:
  1. Count active loans via subquery
  2. Compare with total copies
  3. Determine availability status
Benefits:
  • Efficient single-query operation
  • Real-time availability calculation
  • No need for computed columns
